## 一、Claude Code 的本质定义

### 核心公式

```
Claude Code = 
  上下文背景信息（降低输入任务熵）
  + 工具集（降低执行过程熵）
  + while(true) 循环（持续降熵直到完成）
```

创始人 Boris Cherny 在采访中的原话与之完全一致：

> **"You have a model, you give it tools, you give it some context and a task, and it uses the tools to accomplish the task."**

Claude Code **16 个月来的所有演进，都没有改变这个核心模式**。权限系统、CLAUDE.md、AgentTool、Skill、MCP、Hook、Compact、Plan Mode、Advisor——全是围绕这个循环的优化，不是对它的重构。

### 源码中的核心循环（src/query.ts）

```typescript
while (true) {
  // 发送: system prompt + 上下文 + 历史 + 工具定义 → API
  const response = await callModel({ system, tools, messages })
  
  // 判断: 输出中有没有 tool_use 块？
  if (response.any(block => block.type === "tool_use")) {
    // 模型需要更多信息 → 执行工具 → 注入结果 → 继续
    for (const toolCall of response.toolCalls) {
      const result = executeTool(toolCall)
      messages.push(result)
    }
    continue
  } else {
    // 模型认为任务完成 → 输出文本 → 结束
    displayToUser(response.text)
    break
  }
}
```

## 二、以信息论为框架的解构

### 2.1 香农熵的定义

$$H(X) = -\sum_{i=1}^{n} p(x_i) \log_2 p(x_i)$$

**熵是接收系统对消息不确定度的度量。** 同一个消息对不同的接收系统熵值不同——「商业化测试」对外贸老板是高熵（5种可能平权），对测试专家是低熵（1种概率占优）。

### 2.2 整个系统的本质：降熵管道

```
用户输入（极高熵 H ≈ 3-5 bit）
  │
  ├── 前置上下文收集（第一层降熵）
  │   ├── git status/log → 仓库状态确定化
  │   ├── CLAUDE.md → 项目约定确定化
  │   ├── memdir/ → 持久记忆确定化
  │   └── env info → 运行环境确定化
  │   → 熵降至 H ≈ 1.5-3 bit
  │
  ├── while(true) 循环（逐层降熵直到 H ≈ 0）
  │   每轮迭代:
  │     模型输出 text（与用户通信）
  │        或
  │     tool_use → 执行 Tool → 注入确定信息 → 熵再降低
  │
  └── 停止条件: 模型不再输出 tool_use
      → H ≈ 0（系统认为任务已完成）
```

### 2.3 各组件的熵角色

| 组件 | 信息论角色 | 源码位置 |
|------|-----------|---------|
| **System Prompt** | 先验分布注入（告诉模型「你是什么、该怎么做」） | `src/constants/prompts.ts` (600+行) |
| **CLAUDE.md** | 条件熵 H(输出\|项目知识) 的降低 | `src/utils/claudemd.ts` |
| **memdir/** | 跨会话的持久化先验 | `src/memdir/` |
| **Tool Schema** | 输出空间约束（Zod 校验拒绝非法输出） | 每个工具定义中 |
| **Tool 执行** | 把模型「不确定的事实」变为「确定的信息」 | `src/services/tools/` |
| **Compact** | 在有限上下文预算内保留高价值信息 | `src/services/compact/` |
| **Permission** | 在关键决策点引入人类仲裁 | `src/hooks/toolPermission/` |
| **Skill** | 方法论注入（告诉模型「按这个套路做事」） | `src/skills/bundled/` |
| **Agent** | 独立降熵子循环（隔离上下文运行） | `src/tools/AgentTool/` |
| **Hook** | 外部确定性信号注入（脚本输出喂给模型） | `src/utils/hooks.ts` |
| **Plan Mode** | 执行前先显式化降熵路径 | `src/tools/EnterPlanModeTool/` |
| **Advisor** | 在关键决策点引入更强模型判断 | API 层功能 |

## 三、模型与工具的关系

### 3.1 模型能力是底座，但不是核心壁垒

从泄露版源码可以确认：**Claude Code 不是靠模型微调来适配工具的。**

```typescript
// 模型调用只有一行（src/query.ts:659）
for await (const message of deps.callModel({...}))

// 模型不知道 FileReadTool、BashTool 是什么
// 它只知道: 有个工具叫这个名字, description 说它能读文件, schema 定义了参数
// 这是模型原生具备的 tool use 能力，不是微调来的
```

模型通过 API 的 `tools` 参数接收工具定义（name + description + input_schema），天然就理解怎么用。**工具调用能力是模型在基础训练中就学会的，不是针对 Claude Code 40 多个工具单独微调的。**

### 3.2 Pro vs Flash 的本质差异

| 维度 | Pro（旗舰） | Flash（轻量） | 差距原因 |
|------|------------|-------------|---------|
| HumanEval（函数补全） | ~99% | ~97% | **极小（确定性任务已饱和）** |
| 多步推理（GPQA Diamond） | 94% | 74% | **大（模糊判断需要参数容量）** |
| 错误恢复 | 强（会换思路） | 弱（容易放弃） | 参数规模决定的策略丰富度 |
| 长文档跨章节推理 | 89% | 79% | 注意力维持能力差距 |

**Pro 买的不是「干活的能力」，而是「在信号模糊时做出更好判断的能力」。** 确定性任务（HumanEval）上 Pro 和 Flash 差距趋近于零，因为训练信号足够清晰，小参数也能收敛。

### 3.3 关键推论

> **Flash + 好的任务分解 + 清晰的方法论 ≈ Pro 的效果。**

当任务被充分降熵后，轻量模型的输出质量可以接近旗舰模型。Anthropic 2026年4月发布的 Advisor Strategy 验证了这一点：Haiku 4.5 + Opus Advisor 的效果几乎翻倍（19.7%→41.2%），成本仅为 Sonnet 的 15%。

## 四、编程能力评测体系

### 4.1 评测分层

| 层级 | 代表测试 | 测什么 | 模型饱和状态 |
|------|---------|-------|------------|
| **L1 代码生成** | HumanEval / MBPP | 函数级补全 | **已饱和（99%+）** |
| **L2 软件工程** | SWE-bench Pro / FrontierSWE | bug修复、跨文件修改 | **竞争激烈（60-80%）** |
| **L3 Agent 能力** | Terminal-Bench / MCP-Atlas | 终端操作、工具调用 | **快速提升中** |
| **L4 超长程任务** | SWE-Marathon | 数小时端到端开发 | **差距仍大（13-26%）** |
| **L5 审美/设计** | Design Arena / Code Arena | 前端代码视觉效果 | **格局未定** |

### 4.2 评测的本质：可验证 vs 不可验证

**提升幅度 = 任务的可验证程度。**

| 评测 | 验证方式 | GLM-5.1→5.2 提升 |
|------|---------|-----------------|
| AIME 2026（数学） | 答案确定 | 95.3%→**99.2%** |
| FrontierSWE（有测试） | 测试通过/不通过 | 30.5→**74.4** +44分 |
| Terminal-Bench（结果可验证） | 操作结果匹配 | 62→**81** +19分 |
| CritPt（批判性思维） | 无客观标准 | 4.6→**20.9** 仍然低 |

**确定性任务的训练信号清晰，RL 能高效优化。开放任务的训练信号模糊，模型表现飘忽不定。** 这解释了大模型在确定性任务上趋近满分、在开放任务上力不从心的根本原因。

## 六、系统现存的根本问题

### 6.1「模型判完成」的缺陷

```typescript
// src/query.ts:1062-1357
if (!needsFollowUp) {
  // 模型没有调用工具 → 它认为自己任务完成了
  return { reason: 'completed' }
}
```

**判断标准是「模型没再调用工具」，而不是「任务客观上完成了」。** 这是两个完全不同的东西：

| 模型状态 | 实际结果 | 问题 |
|---------|---------|------|
| "我觉得做完了" → 不调工具 | 任务完成 | ✅ |
| "我觉得做完了" → 不调工具 | 任务没完成 | ❌ 模型过度自信 |
| "我觉得没做完" → 调工具 | 任务已完成 | ❌ 浪费 token |
| "我觉得没做完" → 调工具 | 确实没完成 | ✅ |

**没有客观的"完成标准"。** 这个判断 100% 依赖模型内部无法观测的"信心阈值"。

### 6.2 高熵输入靠模型猜

用户输入一个高熵指令（"做个魂斗罗"、"商业化部署"），系统没有结构化的需求澄清流程，而是让模型自己在推理过程中"猜"用户的真实意图。模型越强猜得越准，但这是在**掩盖架构问题而不是解决问题**。

### 6.3 任务价值分布的失衡

```
价值曲线:

  原始输入（高熵）→ 降熵过程 → 最终输出（低熵）
    价值≈0         值钱        价值≈电费

当前系统把大部分精力投入在「最终输出」的优化上
但真正的价值在「降熵过程」——需求澄清、架构设计、方法论选择
```

## 七、未来演进方向

### 7.1 从「一个大循环」到「循环网络」

```text
现在: 一个模型做所有事
      → 上下文塞爆
      → 既写代码又做架构又测试
      → 一件事卡住全停

未来: 主循环拆任务 + 分派给专业子循环
      ├── 架构循环（Gemini/Opus）
      ├── 实现循环（Sonnet/DeepSeek）
      ├── 测试循环（Haiku）
      └── 审计循环（Opus）
      每个循环独立运行、独立验证、用最适合的模型
```

源码中的 `COORDINATOR_MODE`、`AgentTool`、`WorkflowTool` 已经是这个方向的雏形。

### 7.2 从「模型唯一智能」到「智能分布各层」

```text
现在: 模型是大脑，工具是手脚
      模型每步都要决策

未来: 每个工具内嵌小模型
      工具 → 专家（有方法论、能独立决策）
      主模型 → 协调者（只负责分派和验收）
```

### 7.3 从「自然语言接口」到「结构化接口」

当任务确定性高时直接执行，只有在新领域时才走自然语言降熵流程。可复用的方法论封装为 Skill/Agent，使历史经验不再是「做过就忘」而变为「可持久化、可复用的低熵操作表」。

### 7.4 从「会话记忆」到「知识基础设施」

```text
现在: 每次会话读 CLAUDE.md
      会话后记忆消失

未来: 项目知识图谱持续累积
      按需查询（不是全部塞入上下文）
      模型主动更新知识库
```

### 7.5 从「被动响应」到「主动值守」

Cron Agent 长期值班：定时安全扫描、API 监控自动排查、Issue 自动分析优先级。源码中的 `AGENT_TRIGGERS`、`CRON_TOOLS` 已是这一步的铺垫。

## 八、提出新概念：实时熵值评估系统

### 8.1 当前空白

在整个分析过程中，我们发现了一个**系统性的空白**——没有人在做「实时评估系统自身不确定度」这件事：

| 已有工具 | 测什么 | 实时？ | 能指导下一步？ |
|---------|--------|-------|-------------|
| claude-entropy | token浪费、错误模式 | ❌ 事后 | ❌ |
| claude-hud | 上下文用量、耗时 | ✅ | ❌ 只显示不决策 |
| mete | 代码结构复杂度 | 按需 | ❌ |
| **空白** | **任务信息熵 H(t)** | **✅ 需要** | **✅ 推荐降熵策略** |

### 8.2 系统设计思路

```
实时熵值评估系统的核心组件:

① 熵值评估器
   输入: 当前上下文 + 历史 + 工具执行结果
   输出: H(t) = 当前任务不确定度
         H_dimension[] = 每个维度的剩余不确定度

② 降熵策略推荐
   输入: H_dimension[] + 策略效率历史
   输出: 最优下一步动作（问用户/读文件/搜代码/换策略）

③ 停止条件判定
   输入: H(t) < 阈值
   输出: 任务完成判定（客观的、数据化的、而非模型感觉）
```

### 8.3 一个有价值的实现方向

一个插件/MCP 服务器，实时监控对话状态，计算信息熵，在卡住时推荐最优降熵策略。这与现有工具（claude-hud）的组合可以覆盖完整的「监测-分析-决策」链：

- claude-hud 负责**显示**（已存在，需扩展）
- 熵值评估器负责**分析**（空白，需开发）
- 策略推荐器负责**决策**（空白，需开发）

## 九、核心洞见总结

### 9.1 价值转移

> **代码生成趋近于电费，真正值钱的是「把高熵需求转化为低熵指令」的能力。**

当模型的输出成本趋近于零，价值向上游转移——需求确认、架构设计、知识积累、方法论封装——这些降熵活动才是真正的稀缺资源。

### 9.2 模型能力的本质

> **Pro 和 Flash 在确定性任务上几乎没有区别。Pro 买的是「在模糊信号下做出更好判断」的先验分布，而不是「更强的执行能力」。**

这解释了为什么好的方法论 + 轻量模型可以接近旗舰模型的效果——方法论已经把任务的不确定度降到了足够低。

### 9.3 系统的终极矛盾

> **当前 AI 系统最大的问题不是模型不够强，而是把「什么算完成」这个最高熵的判断，交给了最不稳定的环节（模型推理）。**

一个真正稳固的系统应该把完成标准建立在外部可验证的事实上，而不是模型内部的信心阈值。

### 9.4 下一个阶段

> **2024 之前比推理能力，2025 之后比上下文+工具+物理世界接入。下一个阶段是「编排」——多 Agent 协作、长期值守、混合模型编排、人与 AI 混编团队。**

模型的推理能力已趋近实用阈值。下一步是系统化的能力——不只是「能输出正确答案」，而是「能可靠地完成复杂任务」。
