# 从信息熵与控制论重构 AI Agent 的收敛理论

2026 年初，工程界普遍认为 AI Agent 的演进路线是：**提示词（Prompt） $\rightarrow$ 上下文（Context） $\rightarrow$ 外壳（Harness） $\rightarrow$ 循环（Loop）**。

业界将“Loop Engineering”奉为圭臬，认为只要利用大模型互相 Review（Maker-Checker），再套上一层 DevOps 的自动化定时触发脚本，就能实现 Agent 的自主运行。然而，在真实的、容错率极低的生产环境中（如高并发数据处理或底层协议逆向），这种朴素的循环往往以崩溃和发散告终。

问题的根源在于：**我们一直试图用魔法去规范魔法，却违背了最基本的物理与信息学定律。**

## 第一性原理：大模型的“高熵”陷阱

从香农信息论的视角来看，大模型（LLM）的本质是一个基于概率的自回归文本生成器。它天生就是一个**高熵（高不确定性、高信息量）的噪声源**。

在通信系统中，如果让高熵信息在一个充满噪声的信道中无约束地循环，结果必然是系统崩溃。目前的“大模型循环”恰恰犯了这个致命错误：

1. **裁判与运动员同源：** 用 LLM 去校验另一个 LLM 生成的代码，本质是用高熵系统校验自身噪声。这种所谓的“Maker-Checker”非但无法收敛，反而极易陷入“幻觉正反馈”的死循环。
2. **自然语言梯度的失真：** 试图用大段的自然语言（如 `claude.md`）作为前置约束，指望模型自己体会，这种做法在经历多轮状态转换后，会发生严重的“梯度消失”或偏离。

只要系统内没有引入绝对低熵的外部物理约束，单纯的 Loop 只是在以更快的速度、更大的规模制造不可用的垃圾状态。

## 范式重构：Agent 收敛理论 (Agent Convergence Theory)

要让 Agent 真正落地，必须摒弃“把大模型当人看”的思维惯性，将其还原为控制论中的一个系统组件。由此，我们推导出 **Agent Convergence Theory**。

在这个理论下，**Agent 并不是一个循环系统，而是一个不断降低目标条件熵的反馈控制系统。**

### 1. 重新定义核心组件

- **LLM 不是“任务完成者”**，而是**状态转换器（State Transformer）**。它负责在庞大的可能性空间中生成候选状态。
- **Prompt 不是“输入指令”**，而是**自然语言形式的梯度（Gradient）**。它不是一句话，而是一个完整的优化目标。

### 2. 收敛回路的设计

真正的 Codex 或高可用代码 Agent，其运作逻辑绝非简单的 `Prompt $\rightarrow$ LLM $\rightarrow$ Code`，而是高度类似于机器学习中的梯度下降：

> 初始低熵目标 (Goal) $\rightarrow$ LLM 生成初始状态 ($State_1$) $\rightarrow$ **校验器 (Verifier) 计算误差** $\rightarrow$ 产生确定性约束 (Constraint) $\rightarrow$ LLM 生成下一状态 ($State_2$) $\rightarrow$ 收敛

每一次迭代，绝不是为了让 LLM“再试一次”，而是为了利用校验器返回的绝对确定性信息（如第 135 行类型错误），去抹除目标实现路径上的不确定性（降低条件熵）。

## 工程核心：校验器即产品 (The Verifier is the Product)

在 Agent 收敛理论中，Loop（循环）只是实现方式，真正驱动系统收敛的是 **校验器（Verifier）**。**任务的定义权不再属于最初的提示词，而是属于校验器。** 只要系统状态通过了多级 Verifier 的全部断言，任务即宣告 100% 完成。

这也决定了 AI-Native 架构师的核心职责不再是“调教模型”，而是“设计物理法则”：

| **校验层级** | **传统模式 (高熵 / 易发散)**       | **现代 Agent 架构 (低熵 / 强收敛)**                          |
| ------------ | ---------------------------------- | ------------------------------------------------------------ |
| **执行校验** | 用 LLM 提问：“这段代码能跑吗？”    | 扔进隔离 Docker 容器运行，抓取 Coredump 和 Stack Trace。     |
| **业务逻辑** | 依赖人类肉眼 Review (高延迟、高熵) | 将目标转化为 TDD 自动化测试用例，不亮绿灯绝不合并。          |
| **高阶需求** | 客户反馈：“界面太丑、交互不对”     | 引入“降熵翻译节点”，将主观需求转化为确定的技术约束（如指定 UI 库与组件边距）。 |

在诸如 RPC 签名逆向或并发锁设计的复杂场景中，大模型必定会发散。唯有依靠编译沙盒、依赖注入检查、以及模拟高并发风控降级的真实网络探针，才能构成一张严密的“漏斗”。模型在漏斗内穷举，校验器不断收紧边界，最终逼迫状态坍缩为唯一的正确解。
